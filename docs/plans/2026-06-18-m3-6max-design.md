# M3-6max 设计:N 人桌(2-6)引擎 + 全边池

> 日期:2026-06-18
> 范围:把引擎从 Heads-up 泛化为 N=2..6 通用;实现全边池(side pot)结算。
> 影响:`engine.PlayHand` API、`match` 编排、`store` schema(seat 字段)、`web` 多座位渲染。
> 不在本步:M3 的其它子项(复式发牌、决策质量评分、Ollama)。

---

## 1. 目标

`engine.PlayHand(seats []PlayerSeat, button int, cfg Config, rng, handID)` 支持 2-6 人桌,正确处理:
- 盲注:多人桌 BB 只有一个,SB 只有一个,button 在 BB 左侧(标准)
- 行动顺序:preflop BB 之后的位先动;postflop BB 位置最近左先动(简化:按 seat 顺序,跳过弃牌)
- 多级 all-in 产生的多级边池,每人只赢有资格的部分

---

## 2. 关键设计

### 2.1 API 变化(breaking)

```go
// 改前(Heads-up)
func PlayHand(seats [2]PlayerSeat, button int, cfg Config, rng *rand.Rand, handID int) ([]Event, HandResult)

// 改后(N 人通用)
func PlayHand(seats []PlayerSeat, button int, cfg Config, rng *rand.Rand, handID int) ([]Event, HandResult)
```

`seats []PlayerSeat` 长度 2-6,`button` 是 seat 索引。Event.Seat、HandResult.Winners/FinalStacks 都改为 slice(去掉 `[2]` 固定数组)。

### 2.2 盲注位

多人桌标准规则:
- `button` 位 = 庄家(最后行动 postflop)
- button 左 1(顺时针)= SB
- button 左 2 = BB
- **Heads-up 特例保留**:button = SB(N=2 时 button 左 1 折回自己)

简化:**seat 索引按位置物理顺序排列**,`sb = (button+1) % N`,`bb = (button+2) % N`(N=2 时 sb=button,沿用现有 Heads-up 规则)。

### 2.3 行动顺序

- **Preflop**:BB 后第一个未弃牌位先动 = `(bb+1) % N`,顺时针
- **Postflop**:button 后第一个未弃牌位先动 = `(button+1) % N`,顺时针
- 每轮:`acted[i]` 标记,有人加注后清空其他人的 `acted`(因为又轮到他们了)
- 终止:所有未弃牌玩家都行动过 + 本街下注相等,或只剩 1 个未弃牌

### 2.4 边池(side pot)算法

完整算法:
1. 收集所有未弃牌玩家的本手总投入(`total_contributions[i]`)
2. 按 contribution 升序去重排序,得到一组"层级阈值"
3. 对每两个相邻阈值之间,建立一个 pot:
   - 该层 pot = (阈值差) × (contribution ≥ 该阈值的未弃牌人数)
   - 该层 pot 的赢家集合 = contribution ≥ 该阈值的未弃牌玩家
4. 每层 pot 用 `Evaluate` 比较赢家集合定胜负,平分(余数给顺位最先的)
5. 总赢钱 = 玩家在各层赢的钱之和

举例:A 投 100,B 投 50,C 投 30(都未弃牌):
- 阈值 [0, 30, 50, 100]
- 层 1 [0-30]:3 人各 30 → pot 90,3 人都资格,比牌
- 层 2 [30-50]:2 人(A、B)各 20 → pot 40,A vs B
- 层 3 [50-100]:1 人(A)各 50 → pot 50,A 独拿(无人争)
- 总:pot 90+40+50=180 = A 100+B 50+C 30 ✓

### 2.5 投入记录改造

`handState` 加 `totalContrib [N]int` —— 跨街累积每人总投入(用于边池结算)。每街结束仍把 bets 并入 pot 显示,但**真正的结算是按 totalContrib 算边池**(不是单纯按 pot)。

> 这意味着 `pot` 字段语义微变:它仍然是"累积的已结算筹码",但**最终摊牌/弃牌时不直接发给赢家**,而是走 side pot 算法分摊。

### 2.6 弃牌结算

只剩 1 个未弃牌玩家时,该玩家直接拿走**所有层 pot 中他有资格的部分**。其它"无人争"的层级(只有他一人 contribution 达标)也归他。实质:side pot 算法对单赢家场景自动正确。

## 3. Schema / 落库变化

`hands` 表已有 `p1_hole / p2_hole`,改成存所有 seat 的 hole 牌。两种方案:

**方案 A(推荐):actions 表 seat 字段保留;hands 表加 player_holes JSON 列。**
```sql
ALTER TABLE hands ADD COLUMN player_holes TEXT;  -- JSON: {"0":"Ac Kc","1":"Qd 9s","2":"..."}
-- 兼容老数据:p1_hole/p2_hole 保留
```

`actions` 表 `seat` 已经是 0/1,改成 0-5 即可(无 schema 变化,值范围扩大)。`player_id` 已经按 seat 映射,扩展到 N 个 seat 时由 match 层负责。

`games` 表加 `num_seats INTEGER NOT NULL DEFAULT 2`。

### store / match / web 改造

- `store.RecordGame` 接受 N 个 player(不再硬编码 2)
- `match.Play` 接受 `[]PlayerSpec`(N 个),按顺时针座位入座,button 轮换
- `web` 回放页:牌桌改成圆形/扇形布局,N 个座位围绕;时间轴保持

## 4. 任务分解(预估 8-10 个 task)

1. **engine 泛化**:PlayerSeat/[2]→slice、Street、Event、HandResult 全改 slice;`handState` 加 `totalContrib`。breaking,改测试。
2. **engine setupHand N 人**:盲注位、发底牌(N×2)
3. **engine runStreet N 人**:行动顺序(preflop/postflop)、acted 清空逻辑、终止条件
4. **engine side pot 算法**:`computeSidePots([]int, []bool) []SidePot` + 单测(3 人/4 人/多级 all-in)
5. **engine PlayHand N 人结算**:摊牌时按 side pot 分发;弃牌时单赢家通吃
6. **match 包**:接受 []PlayerSpec,N 人编排,button 轮换
7. **store**:schema 加 num_seats/player_holes,迁移;RecordGame/GetGame N 人化
8. **CLI match**:`--players "deepseek:flash,glm:glm4,..."` 多人
9. **web**:回放页多座位布局
10. **收尾**:跑一局 3 人 RuleBot vs RuleBot vs RuleBot 验证

## 6. 已知风险

| 风险 | 对策 |
|---|---|
| 边池算法复杂,容易出 bug | 多个手算 case 的单测覆盖(2/3/4 人、单/多 all-in、平局) |
| breaking API 改动波及 match/store/web/server/test | 任务 1 集中改类型,跑全部测试;后续任务增量 |
| Heads-up 行为不能退化 | N=2 时所有现有测试必须仍通过(回归) |
| web 多座位布局 | 先做扇形静态布局,动画/视觉留后 |
